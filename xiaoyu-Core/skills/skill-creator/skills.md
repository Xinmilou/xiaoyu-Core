# 技能创建器

用于创建新技能并迭代改进它们的工具。

## 核心流程

创建技能的高级流程：

1. **确定目标** - 明确技能应该做什么以及大致如何实现
2. **编写草稿** - 编写技能的初始版本
3. **创建测试** - 设计几个测试用例并运行
4. **评估结果** - 帮助用户定性和定量评估结果
5. **迭代改进** - 根据反馈重写技能
6. **重复优化** - 直到满意为止

---

## 与用户沟通

注意用户的背景，使用适当的术语：

- "评估"和"基准测试"可以直接使用
- "JSON"和"断言"等术语需要确认用户是否熟悉
- 如有疑问，可以简要解释术语含义

## 创建技能

### 捕获意图

首先理解用户的意图。当前对话可能已经包含用户想要捕获的工作流程。

需要明确的问题：

1. 这个技能应该让AI能够做什么？
2. 什么时候应该触发这个技能？（用户短语/上下文）
3. 期望的输出格式是什么？
4. 是否需要设置测试用例来验证技能？

### 访谈和研究

主动询问边缘情况、输入/输出格式、示例文件、成功标准和依赖项。

检查可用的MCP工具 - 如果对研究有用（搜索文档、查找类似技能、查找最佳实践），可以通过子代理并行研究。

### 编写 SKILL.md

根据用户访谈填写以下组件：

- **name**: 技能标识符
- **description**: 何时触发，做什么。这是主要的触发机制 - 包括技能做什么以及何时使用的具体上下文。
- **compatibility**: 必需的工具、依赖项（可选，很少需要）
- **其余内容**: 具体指令

### 技能结构

```
skill-name/
├── SKILL.md (必需)
│   ├── YAML前置信息 (name, description 必需)
│   └── Markdown指令
└── 捆绑资源 (可选)
    ├── scripts/    - 可执行代码，用于确定性/重复性任务
    ├── references/ - 按需加载到上下文的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

### 渐进式加载

技能使用三级加载系统：

1. **元数据** (name + description) - 始终在上下文中 (~100字)
2. **SKILL.md主体** - 技能触发时在上下文中 (<500行理想)
3. **捆绑资源** - 按需加载（无限制）

**关键模式：**
- 保持SKILL.md在500行以内
- 从SKILL.md中清楚地引用文件，并说明何时读取
- 对于大型参考文件(>300行)，包含目录

### 编写风格

- 使用祈使语气编写指令
- 解释为什么事情重要，而不是使用强制性的"必须"
- 运用心智理论，使技能具有通用性
- 先写草稿，然后用新眼光审视并改进

### 测试用例

编写技能草稿后，提出2-3个现实的测试提示 - 真实用户实际会说的话。

保存测试用例到 `evals/evals.json`：

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示",
      "expected_output": "预期结果描述",
      "files": []
    }
  ]
}
```

## 运行和评估测试用例

将结果放在 `<skill-name>-workspace/` 中，作为技能目录的同级目录。

### 步骤1：同时启动所有运行

对于每个测试用例，在同一轮中启动两个子代理 - 一个有技能，一个没有。

**有技能运行：**
```
执行此任务：
- 技能路径: <path-to-skill>
- 任务: <eval prompt>
- 输入文件: <eval files if any, or "none">
- 保存输出到: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
```

**基线运行**（相同提示，但无技能）：
- 保存到 `without_skill/outputs/`

### 步骤2：运行进行时，起草断言

不要只是等待运行完成 - 利用这段时间起草定量断言。

好的断言是可客观验证的，具有描述性名称。

### 步骤3：运行完成时，捕获计时数据

当每个子代理任务完成时，保存计时数据到 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

### 步骤4：评分、聚合和启动查看器

一旦所有运行完成：

1. **评分每个运行** - 生成 `grading.json`
2. **聚合到基准** - 运行聚合脚本
3. **进行分析** - 读取基准数据并找出模式
4. **启动查看器** - 向用户展示结果

### 步骤5：读取反馈

当用户完成后，读取 `feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少轴标签", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈意味着用户认为没问题。专注于用户有具体抱怨的测试用例。

## 改进技能

这是循环的核心。你已经运行了测试用例，用户已经审查了结果，现在你需要根据他们的反馈改进技能。

### 改进思路

1. **从反馈中概括** - 我们正在尝试创建可以跨许多不同提示使用数百万次的技能。不要只针对示例进行调整。

2. **保持提示精简** - 删除不起作用的内容。

3. **解释原因** - 努力解释你要求模型做每件事的**原因**。今天的LLM很聪明，当给出好的框架时，它们可以超越死板的指令。

4. **寻找重复工作** - 如果所有测试用例都导致子代理编写类似的辅助脚本，这是技能应该捆绑该脚本的强烈信号。

### 迭代循环

改进技能后：

1. 将改进应用到技能
2. 将所有测试用例重新运行到新的 `iteration-<N+1>/` 目录
3. 启动审查器
4. 等待用户审查
5. 读取新反馈，再次改进，重复

继续直到：
- 用户说满意
- 反馈全部为空（一切看起来都好）
- 没有取得有意义的进展

## 描述优化

SKILL.md前置信息中的description字段是决定AI是否调用技能的主要机制。创建或改进技能后，提议优化描述以提高触发准确性。

### 步骤1：生成触发评估查询

创建20个评估查询 - 混合应该触发和不应触发的：

```json
[
  {"query": "用户提示", "should_trigger": true},
  {"query": "另一个提示", "should_trigger": false}
]
```

查询必须真实，是用户实际会输入的内容。

**应该触发**的查询(8-10个)：不同的措辞表达相同意图 - 有些正式，有些随意。

**不应触发**的查询(8-10个)：最有价值的是近似匹配 - 共享关键词或概念但实际上需要不同内容的查询。

### 步骤2：与用户审查

使用HTML模板向用户展示评估集进行审查。

### 步骤3：运行优化循环

告诉用户："这需要一些时间 - 我将在后台运行优化循环并定期检查。"

### 步骤4：应用结果

从JSON输出中获取 `best_description` 并更新技能的SKILL.md前置信息。

## 打包和呈现

检查是否有 `present_files` 工具。如果有，打包技能并向用户呈现.skill文件：

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

打包后，告诉用户生成的 `.skill` 文件路径，以便他们安装。

## 参考文件

`agents/` 目录包含专门子代理的指令：

- `agents/grader.md` - 如何根据输出评估断言
- `agents/comparator.md` - 如何进行盲A/B比较
- `agents/analyzer.md` - 如何分析为什么一个版本胜过另一个

`references/` 目录有额外的文档：
- `references/schemas.md` - evals.json、grading.json等的JSON结构

## 核心循环总结

- 弄清楚技能是关于什么的
- 起草或编辑技能
- 在测试提示上运行有技能的AI
- 与用户一起评估输出：
  - 创建benchmark.json并运行评估查看器
  - 运行定量评估
- 重复直到你和用户满意
- 打包最终技能并返回给用户

请将步骤添加到你的待办列表，以确保不会忘记。
